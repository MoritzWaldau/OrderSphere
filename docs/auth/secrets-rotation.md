# Secrets Rotation Procedure

All secrets in non-dev environments are stored in Azure Key Vault (`ordersphere-kv`) and injected at deployment time via Aspire parameters. Dev values are in AppHost user-secrets (not source-controlled).

## Dev Setup (first-time)

Populate AppHost user-secrets with the dev placeholder values:

```sh
dotnet user-secrets set "Parameters:keycloak-admin-password"     "admin"                                         --project src/OrderSphere.AppHost
dotnet user-secrets set "Parameters:bff-client-secret"           "web-bff-dev-secret-change-in-prod"             --project src/OrderSphere.AppHost
dotnet user-secrets set "Parameters:ordering-worker-secret"      "ordering-worker-dev-secret-change-in-prod"     --project src/OrderSphere.AppHost
dotnet user-secrets set "Parameters:notification-worker-secret"  "notification-worker-dev-secret-change-in-prod" --project src/OrderSphere.AppHost
dotnet user-secrets set "Parameters:payment-worker-secret"       "payment-worker-dev-secret-change-in-prod"      --project src/OrderSphere.AppHost
```

## Client Secret Rotation (BFF, ordering-worker, notification-worker)

1. **Generate a new secret** in Keycloak Admin:
   - Navigate to **Clients → {client-id} → Credentials → Regenerate**.
   - Or via admin REST API:
     ```sh
     curl -X POST \
       -H "Authorization: Bearer $ADMIN_TOKEN" \
       https://auth.ordersphere.example.com/admin/realms/ordersphere/clients/{client-uuid}/client-secret
     ```

2. **Update Key Vault**:
   ```sh
   az keyvault secret set \
     --vault-name ordersphere-kv \
     --name bff-client-secret \        # or ordering-worker-secret / notification-worker-secret
     --value "<new-secret>"
   ```

3. **Redeploy** the affected service (BFF or worker). Aspire / Container Apps / Kubernetes picks up the new Key Vault secret version on the next deployment.
   - BFF and workers are stateless with respect to client secrets — there is no grace period needed on the application side.
   - The old secret is invalid in Keycloak immediately after regeneration, so deploy promptly to avoid a gap.

## Keycloak Admin Password Rotation

1. Change via Keycloak admin REST API (requires a current admin token):
   ```sh
   curl -X PUT \
     -H "Authorization: Bearer $ADMIN_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"type":"password","value":"<new-password>","temporary":false}' \
     https://auth.ordersphere.example.com/admin/realms/master/users/{admin-user-uuid}/reset-password
   ```

2. Update Key Vault:
   ```sh
   az keyvault secret set \
     --vault-name ordersphere-kv \
     --name keycloak-admin-password \
     --value "<new-password>"
   ```

3. Redeploy the Keycloak container (or update the environment variable in the running instance without restart if the container runtime supports it).

## DataProtection Key Rotation (.NET session encryption)

DataProtection keys encrypt the session tickets stored in Redis by `RedisTicketStore`. They are **automatically rotated every 90 days** by the .NET runtime. Old keys are retained for decryption until their expiry; no manual action is normally required.

**Forced manual rotation** (e.g., after suspected key compromise):

1. Delete the DataProtection key ring from Redis:
   ```sh
   redis-cli DEL DataProtection-Keys
   ```
   **Effect**: All active BFF sessions are immediately invalidated. Users are signed out and must log in again.

2. The BFF generates a new key ring on the next startup or the first request that requires DataProtection.

**Key persistence location**: Redis key `DataProtection-Keys` (string, XML payload). Protected by the underlying OS Data Protection API on the host; no additional encryption is applied at the Redis layer.

## JWT Signing Key Rotation (Keycloak)

Managed automatically by Keycloak's key provider priority mechanism. No manual procedure is required under normal operation.

- New key pairs are generated by Keycloak when the provider's `priority` increments or the key reaches its maximum age.
- All public keys remain in the JWKS endpoint (`/realms/ordersphere/protocol/openid-connect/certs`) for the `rotatedKeyExpiry` period (default: 30 days), so services that cache JWKS continue to validate tokens issued with the old key.
- Services using `AddOrderSphereJwtAuth()` refresh JWKS automatically every 10 minutes, ensuring new signing keys are picked up without a restart.

**Forced rotation**: In Keycloak Admin, navigate to **Realm Settings → Keys → rsa-generated → Rotate** and confirm.

## Audit Trail

All secret rotation events should be recorded in the team's change log (e.g., Jira, Confluence). For Azure Key Vault operations, enable **Key Vault diagnostic logging** to Log Analytics to capture all read/write events on secrets.
